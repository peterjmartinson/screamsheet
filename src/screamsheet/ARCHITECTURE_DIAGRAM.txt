DATA FLOW
=========

1. User creates screamsheet via Factory:
   ScreamsheetFactory.create_mlb_screamsheet(...)

2. Factory instantiates appropriate screamsheet class:
   MLBScreamsheet → creates MLBDataProvider

3. Screamsheet builds sections:
   build_sections() → [GameScoresSection, StandingsSection, ...]

4. Each section:
   - Fetches data from provider: fetch_data()
   - Renders to PDF elements: render()

5. Screamsheet generates PDF:
   generate() → assembles all sections into PDF

6. PDF saved to file


EXTENSIBILITY POINTS
====================

Add New Sport:
1. Create XDataProvider(DataProvider)
2. Create XScreamsheet(SportsScreamsheet)
3. Add factory method (optional)
4. Done!

Add New News Source:
1. Create XProvider
2. Create XScreamsheet(NewsScreamsheet)
3. Add factory method (optional)
4. Done!

Add Custom Section:
1. Create XSection(Section)
2. Add to build_sections()
3. Done!


DESIGN PATTERNS USED
====================

1. Factory Pattern: ScreamsheetFactory
2. Template Method: BaseScreamsheet.generate()
3. Strategy Pattern: DataProvider implementations
4. Composite Pattern: Sections compose screamsheets
5. Inheritance: Class hierarchy for reuse
6. Separation of Concerns: Clear boundaries


KEY PRINCIPLES
==============

✓ Single Responsibility: Each class has one job
✓ Open/Closed: Open for extension, closed for modification
✓ Liskov Substitution: Subclasses are substitutable
✓ Interface Segregation: Small, focused interfaces
✓ Dependency Inversion: Depend on abstractions
✓ DRY: Don't Repeat Yourself
✓ KISS: Keep It Simple, Stupid
✓ Composition over Inheritance: Flexible section composition
